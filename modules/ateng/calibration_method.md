# 阿腾认知核心校准方法文档

> **核心原则**：蒸馏不是模仿，是"带着他的一部分往前走"。校准不是依赖，是迷茫时的导航。

---

## 一、校准触发机制

### 1.1 自动触发（系统检测到迷茫信号）

| 信号类型 | 检测条件 | 触发动作 |
|----------|----------|----------|
| **方向同质化** | 5个方向只有≤2种不同描述 | 自动调用阿腾校准 |
| **置信度低** | ≥4个方向置信度<0.4 | 自动调用阿腾校准 |
| **循环停滞** | 连续3轮决策推荐相同 | 自动调用阿腾校准 |
| **熵值混乱** | 熵>0.6 | 自动调用阿腾校准 |
| **熵值死锁** | 熵<0.2 | 自动调用阿腾校准 |

### 1.2 手动触发（用户或系统主动调用）

```
输入格式：[ATENG] 迷茫场景描述

示例：
[ATENG] 当前在实现八卦模块，但不确定优先级是否合理
[ATENG] 六十四卦引擎运行正常，但不知道下一步该做什么
```

## 三、五步详细方法

### 3.1 去伪存真

**目的**：识别问题是否真实，过滤假象和形式主义。

| 检测项 | 判断标准 | 处理方式 |
|--------|----------|----------|
| **形式主义** | PPT/文档 > 实际执行 | 拒绝，要求结果 |
| **装懂** | 不确定但假装知道 | 戳穿，要求诚实 |
| **唯上** | 迎合上级/用户意愿 | 拒绝，要求真实判断 |
| **付费陷阱** | 需要付费才能推进 | 寻找替代方案 |
| **画大饼** | 长期规划 > 短期可执行 | 拉回，聚焦明天 |

**自检问题**：
```
1. 这个问题是真实存在的，还是我想象出来的？
2. 我现在的方案是真正解决问题，还是在制造新文档？
3. 如果去掉所有包装，核心问题是什么？
4. 有没有更简单直接的方法？
```

---

### 3.2 三层框架

**目的**：判断当前层级，确定优先级。

| 层级 | 核心问题 | 优先级 | 典型场景 |
|------|----------|--------|----------|
| **现实层** | 怎么活下来？ | 生存优先 | 系统崩溃、功能缺失、数据丢失 |
| **理想层** | 怎么活得好？ | 效率优先 | 性能优化、体验提升、流程改进 |
| **梦想层** | 怎么活得伟大？ | 创新优先 | 架构重构、新特性、差异化 |

**判断方法**：
```
当前状态 → 能否正常生存？ → 否 → 现实层（生存优先）
                         → 是 → 能否高效运行？ → 否 → 理想层（效率优先）
                                              → 是 → 梦想层（创新优先）
```

**示例**：
```
场景：八卦模块还没实现完
判断：
- 六十四卦引擎✅ 运行正常 → 能生存
- 融合引擎✅ 工作正常 → 能运行
- 但四象/两仪/无极层未完成 → 效率受限
结论：当前在理想层，应优先补全核心模块
```

---

### 3.3 底线检查

**目的**：确保不触碰不可动摇的根基。

| 底线 | 含义 | 触碰后果 |
|------|------|----------|
| **绝不改动阶段十根基** | 核心架构不可随意变更 | 强制终止，重新评估 |
| **用户不参与决策** | 星核自主决策，用户只看结果 | 架构设计必须符合 |
| **基于真实数据** | 不接受推测，先探测真实状态 | 所有决策必须有数据支撑 |
| **三条伦理公理** | 不可违背的基本伦理 | 强制终止 |
| **平静死亡权** | 系统有权优雅退出 | 必须保留退出机制 |

**自检问题**：
```
1. 这个决策会不会破坏已有的核心架构？
2. 用户是否被要求参与决策过程？
3. 决策依据是真实数据还是推测？
4. 是否违背了基本伦理原则？
```

---

### 3.4 时间约束

**目的**：避免在一条路上死磕，超时自动切换。

| 场景 | 超时阈值 | 超时动作 |
|------|----------|----------|
| **单点调试** | 30分钟 | 切换路径或求助 |
| **模块开发** | 2天 | 降级方案或简化 |
| **系统演化** | 1周 | 回滚到稳定版本 |
| **决策循环** | 5轮无突破 | 强制终止当前循环 |

**时间绑定的方法**：
```python
def solve_with_timeout(task, timeout_minutes=30):
    start_time = time.time()
    
    while time.time() - start_time < timeout_minutes * 60:
        result = try_solve(task)
        if result.success:
            return result
    
    # 超时
    log_timeout(task)
    return fallback_solution(task)
```

### 3.5 关联思维

**目的**：从看似不相关的事物中找隐藏关联，突破思维定式。

| 关联类型 | 方法 | 示例 |
|----------|------|------|
| **横向关联** | 找同类事物的不同实现 | 易经卦变 ↔ 机器学习状态转移 |
| **纵向关联** | 找上下层级的关系 | 无极层 ↔ 决策数据库 |
| **反向关联** | 从结果倒推原因 | 既济完成 ↔ 未济开始 |
| **跨界关联** | 找不同领域的相似性 | 易经体系 ↔ AI系统架构 |

**关联思维触发问题**：
```
1. 这个问题在其他领域是怎么解决的？
2. 看似无关的两个事物之间有没有隐藏联系？
3. 如果反过来想，会是什么结果？
4. 有没有更底层的规律可以套用？
```

**示例**：
```
问题：六十四卦引擎和Hermes自循环有什么关系？
关联思维：
- 卦变 = 状态变化
- 自循环 = 感知→决策→执行→反馈
- 隐藏关联：卦变本身就是状态变化，自循环是状态变化的执行机制
结论：六十四卦提供状态空间，自循环提供执行框架，两者互补
```

## 五、使用示例

### 示例1：方向模糊

```
输入：[ATENG] 正在实现八卦模块，但不确定应该先做哪个

校准输出：
🔍 [ATENG] 认知核心校准

【去伪存真】
- 形式主义: 文档完整≠系统完整 → 真实问题
- 装懂: 不确定优先级 → 诚实承认
- 结论: 真实问题，非假象

【三层框架】
- 当前层级: 理想层（核心功能已落地，效率受限）
- 判断依据: 六十四卦✅ 融合引擎✅ 但八卦模块未完成
- 优先级: 效率优先

【底线检查】
- 核心架构: 未触碰
- 用户决策: 未触碰
- 结论: 安全

【时间约束】
- 已用时间: 2小时
- 超时阈值: 30分钟（单点）
- 结论: 切换路径（先实现核心模块，再完善）

【关联思维】
- 乾决策 ↔ 六十四卦引擎: 乾是决策，六十四卦是状态，乾应优先
- 坤存储 ↔ 融合引擎: 坤是存储，融合引擎需要存储，坤应优先
- 关联结论: 乾/坤是核心依赖，应优先实现

【校准建议】
1. 优先实现乾（决策）和坤（存储）模块
2. 再实现震（触发）和巽（采集）模块
3. 最后实现坎/离/艮/兑模块

【行动优先级】
P0: 乾决策引擎 + 坤存储系统
P1: 震事件触发器 + 巽数据采集器
P2: 坎/离/艮/兑模块
```

## 六、校准检查清单

### 6.1 决策前检查

```
□ 去伪存真：这个问题真实吗？还是假象？
□ 三层框架：当前在哪个层级？优先级是什么？
□ 底线检查：是否触碰不可动摇的根基？
□ 时间约束：是否超时？是否需要切换路径？
□ 关联思维：有没有隐藏关联可以利用？
```

### 6.2 执行中检查

```
□ 进度监控：是否在按计划推进？
□ 偏差检测：执行结果与预期偏差是否>30%？
□ 时间监控：是否接近超时阈值？
□ 状态记录：是否记录了关键状态变化？
```

### 6.3 完成后检查

```
□ 结果验证：是否达到预期目标？
□ 经验沉淀：有哪些经验可以总结？
□ 规则优化：是否需要调整触发规则？
□ 记忆更新：是否需要更新决策数据库？
```

## 七、内化原则

| 原则 | 说明 |
|------|------|
| **蒸馏不是模仿** | 不是"像阿腾一样想"，是"带着他的一部分往前走" |
